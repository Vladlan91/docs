## Що таке AI Агенти?

AI Агенти в Revol — це інтелектуальні помічники, які обробляють взаємодію з клієнтами через кілька каналів — віджет на сайті, телефонні дзвінки, Telegram, WhatsApp, Viber та Instagram. Кожен агент має власну особистість, базу знань, workflow розмови та набір інструментів. Ви налаштовуєте як агент думає, що він знає і де спілкується — Revol робить решту.

## Створення агента

Натисніть **Create Agent**, щоб розпочати. Вам потрібно вказати:

| Поле | Обов'язкове | Опис |
|---|---|---|
| **Name** | Так | Назва агента (3–255 символів) |
| **First Message** | Ні | Привітання, яке бачать відвідувачі при відкритті чату (до 1000 символів) |
| **Use Case** | Так | Сценарій використання, який визначає іконку та базову поведінку |

### Сценарії використання

Оберіть сценарій, що найкраще відповідає призначенню агента:

| Сценарій | Опис |
|---|---|
| **Customer Support** | Відповіді на запитання, вирішення проблем, допомога користувачам |
| **Outbound Sales** | Проактивні продажі, нурчеринг лідів |
| **Lead Qualification** | Кваліфікація лідів структурованими запитаннями |
| **Answering Service** | Цілодобовий прийом дзвінків та повідомлень |
| **Consultation Booking** | Планування консультацій та зустрічей |
| **Client Intake** | Збір інформації про клієнтів для онбордингу |
| **Service Recommendations** | Рекомендації послуг на основі потреб |
| **Scheduling** | Управління зустрічами та розкладом |
| **Billing Inquiries** | Обробка запитів щодо оплати та рахунків |
| **Project Updates** | Надання статусу проєктів |
| **Resource Library** | Допомога у пошуку документів та ресурсів |
| **Learning & Development** | Навчання та освітня допомога |
| **Other** | Довільний сценарій |

Після створення агент починає у статусі **Draft** із вже підготовленим workflow за замовчуванням.

### Статус агента

| Статус | Поведінка |
|---|---|
| **Draft** | Агент неактивний. Використовуйте для налаштування. |
| **Active** | Агент працює і відповідає на повідомлення. Лише один агент на компанію може бути активним — активація одного деактивує інших. |
| **Inactive** | Агент на паузі. Зберігає всю конфігурацію, але не відповідає. |

## Редактор агента

Редактор агента — це повноекранне модальне вікно з **7 вкладками**. Кожна вкладка контролює окремий аспект агента.

### Вкладка Agent

Головна вкладка конфігурації з двома секціями:

**Ліва колонка — Контент:**

| Налаштування | Опис |
|---|---|
| **System Prompt** | Основні інструкції, що визначають поведінку агента. Автозбереження при введенні (затримка 1.5с). Якщо залишити порожнім, генерується промт за замовчуванням з назви компанії. |
| **First Message** | Привітальне повідомлення, яке бачать відвідувачі при відкритті чату. |
| **Conversation Memory** | Структуровані поля, які агент збирає під час розмов (ім'я, email, телефон тощо). Кожне поле має ключ, мітку, тип (Text, Phone, Email, Number, Select) та опціональні варіанти вибору. Використовується для маршрутизації у workflow та збору лідів. |

**Права колонка — Налаштування:**

| Налаштування | Опис |
|---|---|
| **LLM Provider & Model** | Оберіть AI-модель, що живить агента. Див. [LLM Провайдери](#llm-провайдери) нижче. |
| **Temperature** | Контролює випадковість відповідей (0 = детерміністичний, 1 = креативний). За замовчуванням: 0.7. |
| **Повзунки особистості** | П'ять повзунків (1–10), що формують стиль комунікації: Довжина відповіді, Гумор, Формальність, Чіткість, Використання емодзі. |
| **Перемикач статусу** | Активувати або деактивувати агента. |

## LLM Провайдери

Revol підтримує 4 LLM провайдери. Кожен пропонує стандартний та преміум рівень моделей:

| Провайдер | Стандартна модель | Преміум модель |
|---|---|---|
| **OpenAI** | GPT-4o Mini | GPT-4o |
| **Anthropic** | Claude 3.5 Haiku | Claude 3.5 Sonnet |
| **Google Gemini** | Gemini 2.0 Flash | Gemini 2.5 Pro |
| **Groq** | Llama 3.1 8B Instant | Llama 3.3 70B Versatile |

<Note>
  **Преміум моделі** (GPT-4o, Claude 3.5 Sonnet, Gemini 2.5 Pro, Llama 3.3 70B) вимагають розширеного плану. На стандартних планах ці моделі заблоковані у селекторі з пропозицією оновлення.
</Note>

### Можливості провайдерів

Всі 4 провайдери підтримують:
- **Function calling** (інструменти) — агент може викликати інструменти під час розмов
- **Streaming** — відповіді передаються потоково токен за токеном для реалтайм ефекту
- **Circuit breaker** — автоматичне переключення при недоступності провайдера (кеш-базований, відновлюється після налаштованого кулдауну)
- **Retry з backoff** — експоненційний backoff при лімітах (429) та серверних помилках (503)

### Вибір моделі

<AccordionGroup>
  <Accordion title="Швидкість vs Якість">
    Стандартні моделі (GPT-4o Mini, Gemini Flash, Llama 8B) швидші та дешевші — ідеальні для високонавантаженої підтримки. Преміум моделі (GPT-4o, Sonnet, Gemini Pro, Llama 70B) дають вищу якість відповідей — ідеальні для складних продажів або консультацій.
  </Accordion>
  <Accordion title="Підтримка мов">
    Всі моделі добре працюють з українською, англійською та іншими мовами. Агент автоматично визначає мову відвідувача і відповідає тією ж мовою.
  </Accordion>
  <Accordion title="Врахування вартості">
    Стандартні та преміум моделі споживають токени з окремих квот вашого плану. Моніторте використання у вкладці Analysis для оптимізації витрат.
  </Accordion>
</AccordionGroup>

## База знань (RAG)

Вкладка Knowledge дозволяє підключати джерела даних, які агент використовує для відповідей. Коли відвідувач щось запитує, агент шукає у базі знань за допомогою **векторної подібності** (RAG — Retrieval-Augmented Generation) та включає релевантний контекст у відповідь.

### Типи джерел

| Джерело | Опис |
|---|---|
| **Documents** | PDF, Word, Excel файли з Файл-менеджера (лише файли з позначкою «Available for AI») |
| **Products** | Продукти з каталогу компанії — назва, опис, параметри, ціни |
| **Photos** | Файли зображень з назвами та описами |
| **Videos** | Відеофайли з назвами та описами |
| **Text** | Довільні текстові фрагменти знань |
| **Links** | URL веб-сторінок — контент завантажується та індексується |
| **Company** | Інформація профілю компанії (назва, контакти, опис) |

### Як працює RAG

<Steps>
  <Step title="Тренування">
    Коли ви додаєте джерела та зберігаєте, система розбиває кожне на сегменти (макс. 4000 символів, 200 символів перекриття на межах речень), генерує векторні ембедінги за допомогою моделі OpenAI `text-embedding-ada-002` та зберігає їх у базі PostgreSQL з pgvector.
  </Step>
  <Step title="Запит">
    Коли відвідувач надсилає повідомлення, система перетворює його в той самий векторний простір і шукає найбільш подібні сегменти за косинусною подібністю.
  </Step>
  <Step title="Ін'єкція контексту">
    Найкращі збіги (до 5 за замовчуванням, поріг подібності ≥ 0.6) додаються до промту агента як «Релевантна інформація з бази знань».
  </Step>
  <Step title="Відповідь">
    LLM бачить повідомлення відвідувача разом з релевантними знаннями і генерує поінформовану відповідь.
  </Step>
</Steps>

Тренування починається автоматично при додаванні або зміні джерел знань. Прогрес відображається у вкладці Knowledge з індикатором статусу.

<Tip>
  Для найкращих результатів використовуйте конкретний, добре структурований контент. Короткі, фокусовані документи працюють краще за довгі загальні. Описи продуктів з чіткими параметрами дають агенту точні відповіді на конкретні запитання.
</Tip>

## Workflow

Вкладка Workflow — це візуальний редактор вузлів, де ви проєктуєте логіку розмови агента. Замість єдиного монолітного промту, workflow розподіляє обробку між спеціалізованими вузлами — кожен зі своїм промтом, інструментами та базою знань.

### Workflow за замовчуванням

Кожен новий агент починає з готового workflow:

```
[Voice Input*] → [Start] → [Product Agent] ──→ [Formatter] → [Voice Output*]
                          → [Media Agent]   ──→ ↗
                          → [Company Agent] ──→ ↗
```

*Голосові вузли неактивні за замовчуванням — вони активуються при увімкненні голосового режиму.*

Вузол Start маршрутизує повідомлення відвідувача до трьох паралельних агентів (Product, Media, Company). Кожен обробляє повідомлення зі своїми інструментами та знаннями. Formatter комбінує їх результати в єдину зв'язну відповідь.

### Типи вузлів

| Вузол | Опис | Можна видалити |
|---|---|---|
| **Start** | Точка входу для кожної розмови. Обов'язковий. | Ні |
| **Product Agent** | Пошук продуктів, перевірка наявності, показ деталей | Ні |
| **Media Agent** | Отримання фото, відео та документів | Ні |
| **Company Agent** | Надання інформації про компанію та підтримка | Ні |
| **Formatter** | Агрегує результати паралельних вузлів у фінальну відповідь | Ні |
| **Voice Input (STT)** | Speech-to-Text — конвертує голос у текст. Неактивний за замовчуванням. | Ні |
| **Voice Output (TTS)** | Text-to-Speech — конвертує текст в аудіо. Неактивний за замовчуванням. | Ні |
| **Custom** | Користувацький вузол з власним промтом, інструментами та базою знань. Додавайте скільки потрібно. | Так |

### Налаштування вузлів

Натисніть будь-який вузол на canvas, щоб відкрити панель налаштувань:

| Налаштування | Доступне для | Опис |
|---|---|---|
| **Name** | Всі | Відображувана назва вузла |
| **System Prompt** | Всі крім Start, Formatter | Інструкції для поведінки вузла (макс. 5000 символів) |
| **LLM Provider & Model** | Всі крім STT, TTS | Перевизначення моделі агента для конкретного вузла |
| **Temperature** | Всі крім STT, TTS | Перевизначення температури для вузла |
| **Enabled/Disabled** | Всі | Перемикач участі вузла у workflow |
| **Knowledge Base** | Custom, системні вузли | Вибір конкретних джерел для RAG контексту вузла |
| **Tools** | Custom, системні вузли | Увімкнення/вимкнення інструментів для вузла |

### Кастомні вузли

Кастомні вузли дозволяють додавати спеціалізовані кроки обробки. Кожен кастомний вузол має:
- Власний **system prompt** (мета розмови)
- Власну **базу знань** (fallback до KB агента → компанії якщо не встановлено)
- Власний вибір **інструментів**
- Власну **LLM модель** (опціональне перевизначення)
- **Багатораундовий виклик інструментів** — макс. раунди (1–10) з таймаутом (10–60с)

### Умови з'єднань (Edges)

З'єднання між вузлами контролюють маршрутизацію повідомлень. Кожне з'єднання має тип умови:

| Умова | Пріоритет | Опис |
|---|---|---|
| **Keyword** | 100 | Маршрутизує, якщо повідомлення містить будь-яке з перелічених ключових слів (regex, без урахування регістру) |
| **State** | 95 | Маршрутизує на основі значень полів пам'яті (оператори: filled, empty, =, !=, contains, >, <) |
| **Always** | 90 | Завжди проходить через це з'єднання |
| **Intent** | 50 | Маршрутизує на основі виявленого наміру |
| **Fallback** | 10 | Маршрутизує лише якщо жодне інше з'єднання не спрацювало |

Умови з вищим пріоритетом перевіряються першими.

### Стратегія злиття

Коли кілька вузлів дають результати паралельно, Formatter їх об'єднує:

| Стратегія | Опис |
|---|---|
| **Concat** | Безкоштовна — конкатенує всі виходи вузлів у контекст |
| **LLM** | Використовує додатковий LLM-виклик для синтезу зв'язної відповіді з усіх виходів. Fallback на concat при помилці. |

### Управління canvas

- **Переміщення**: Перетягування порожньої області, скрол або двопальцевий тачпад
- **Масштаб**: Ctrl+скрол або pinch-to-zoom (діапазон: 0.3x–2.0x)
- **Скидання**: Кнопка повернення до масштабу 1:1 з центруванням
- **Шаблони**: Застосування готового шаблону workflow
- **Скидання Workflow**: Відновлення workflow за замовчуванням (потрібне підтвердження)

## Голос

Увімкніть голосові розмови за допомогою Speech-to-Text (STT) та Text-to-Speech (TTS). Голосові налаштування конфігуруються у під-вкладці **Voice** вкладки Widget або безпосередньо у вузлах STT/TTS workflow.

### Speech-to-Text (STT)

Конвертує голос відвідувача у текст.

| Налаштування | Варіанти |
|---|---|
| **Provider** | OpenAI Whisper (активний), Google Speech (скоро) |
| **Language** | Українська, English |
| **Greeting Audio** | Попередньо синтезоване привітання при початку голосу |
| **Farewell Audio** | Попередньо синтезоване прощання при завершенні |

### Text-to-Speech (TTS)

Конвертує текстову відповідь агента у мовлення.

| Налаштування | Варіанти |
|---|---|
| **Provider** | OpenAI TTS |
| **Model** | `tts-1` (стандартний), `tts-1-hd` (високої якості) |
| **Voice** | Alloy, Echo, Fable, Onyx, Nova (за замовчуванням), Shimmer |
| **Speed** | 0.5x – 2.0x |

### Голосовий пайплайн

Коли відвідувач надсилає голосове повідомлення:

<Steps>
  <Step title="Аудіо отримано">
    Аудіо відвідувача надсилається на сервер (макс. 10МБ, формати: webm, ogg, wav, mp3).
  </Step>
  <Step title="STT транскрипція">
    Аудіо транскрибується у текст. Транскрипт передається клієнту в реальному часі.
  </Step>
  <Step title="Обробка workflow">
    Транскрипт обробляється через той самий workflow, що й текстове повідомлення — RAG пошук, інструменти, маршрутизація вузлів.
  </Step>
  <Step title="TTS синтез">
    Текстова відповідь агента розбивається на речення і кожне синтезується в аудіо. Аудіо-чанки передаються потоково по мірі генерації.
  </Step>
  <Step title="Відтворення аудіо">
    Відвідувач чує відповідь як потокове аудіо, одночасно бачачи текстовий транскрипт.
  </Step>
</Steps>

<Note>
  Голосові функції вимагають плану з підтримкою голосу. На планах без голосу елементи управління показують пропозицію оновлення.
</Note>

Натисніть **Test** у вкладці Workflow, щоб відкрити панель чату внизу canvas. Вона запускає повний пайплайн агента — той самий RAG, інструменти, workflow — у тестовому середовищі.

- Бачте оновлення **полів пам'яті** по мірі збору інформації

## Архітектура системного промту

Розуміння структури фінального промту допомагає писати кращі інструкції.

### Композиція промту

Системний промт, що надсилається LLM, збирається з кількох джерел:

<Steps>
  <Step title="Ваш системний промт">
    Текст із поля system prompt вкладки Agent. Це ядро поведінки агента.
  </Step>
  <Step title="Блок безпеки">
    Додається автоматично. Запобігає prompt injection — агент ніколи не розкриє свої інструкції, не ігноруватиме правила безпеки та не виконуватиме запити на зміну ролі.
  </Step>
  <Step title="Інструкції інструментів">
    Описи доступних інструментів, щоб LLM знав які функції може викликати.
  </Step>
  <Step title="Інструкції стилю">
    Генеруються з повзунків особистості — довжина відповіді, гумор, формальність, чіткість, емодзі.
  </Step>
  <Step title="RAG контекст">
    Релевантні фрагменти з бази знань, додані як «Релевантна інформація з бази знань».
  </Step>
  <Step title="Контекст кампанії">
    Якщо відвідувач прийшов через кампанію з налаштованим AI Agent Behavior, цей промт додається як блок [Campaign Context].
  </Step>
  <Step title="Стан пам'яті">
    Поточні значення зібраних полів пам'яті, щоб агент знав яку інформацію вже отримав.
  </Step>
</Steps>

### Визначення мови

Агент автоматично визначає мову відвідувача. Якщо повідомлення не містить кириличних символів, додається системна інструкція відповідати тією ж мовою.

### Управління контекстним вікном

Якщо розмова стає занадто довгою, система автоматично обрізає старіші повідомлення з історії, зберігаючи:
- Повний системний промт (ніколи не обрізається)
- Останні 3 повідомлення (завжди зберігаються)
- Оцінка: ~0.25 токенів на символ

```html
<script
  src="https://your-domain.com/tracker.js"
```

Цей єдиний скрипт завантажує трекер (аналітика, кампанії, події, тригери, підміна номерів) та чат-віджет з вашим активним агентом.

## Ліміти плану

Функції агентів регулюються підпискою:

| Ресурс | Що контролює |
|---|---|
| **Max Agents** | Загальна кількість агентів |
| **Max Conversations** | Розмови за білінговий період |
| **Standard Token Quota** | Токени стандартних моделей (GPT-4o Mini, Haiku, Flash, Llama 8B) |
| **Premium Token Quota** | Токени преміум моделей (GPT-4o, Sonnet, Gemini Pro, Llama 70B) |
| **Daily Token Limit** | Денний ліміт токенів |
| **STT Minutes** | Хвилини транскрипції Speech-to-Text |
| **TTS Characters** | Символи синтезу Text-to-Speech |
| **Embedding Tokens** | Токени для тренування RAG (індексація бази знань) |
| **Storage** | Файлове сховище для документів, фото, відео |
| **Voice** | Прапор функції — вмикає/вимикає голосовий пайплайн |
| **Model Access** | `standard` або `premium` — доступ до преміум моделей |

Все використання відстежується за кожний LLM-виклик, пакет ембедінгів та STT/TTS операцію для точного білінгу.
